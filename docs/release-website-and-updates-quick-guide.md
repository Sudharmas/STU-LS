# STU-LS New User Guide: How Releases and In-App Updates Work

This guide explains exactly what to click, what to configure, and how updates reach user laptops.

## 0. First Understand the Flow
1. When you push code to GitHub, users do not get updates automatically.
2. Users get updates only after you create a new GitHub Release with new installer/update files.
3. App checks the updater URL, sees new version, downloads signed update, and installs.

So the real release flow is:
Push code -> Build installers -> Create GitHub Release -> App users click Check for Updates.

## 1. One-Time Setup

### Step 1: Connect app to your GitHub repo for updates
1. Open [apps/desktop/src-tauri/tauri.conf.json](apps/desktop/src-tauri/tauri.conf.json).
2. In updater endpoints, set:
3. https://github.com/OWNER/REPO/releases/latest/download/latest.json
4. Replace OWNER and REPO with your real GitHub repo.

Why:
1. This is where app checks for the newest release metadata.

### Step 2: Create updater signing keys
1. Open terminal in project root.
2. Run: npm run tauri signer generate
3. Save private key safely (password manager, vault, secure drive).
4. Copy public key into [apps/desktop/src-tauri/tauri.conf.json](apps/desktop/src-tauri/tauri.conf.json) at plugins.updater.pubkey.

Why:
1. App installs only signed updates.
2. This prevents malicious fake updates.

### Step 3: Keep private key available during build
1. Before release build, set environment variable TAURI_SIGNING_PRIVATE_KEY.
2. If your key has password, set TAURI_SIGNING_PRIVATE_KEY_PASSWORD.

Why:
1. Tauri needs this private key to generate valid update signatures.

## 2. First Release (Click-by-Click)

### Step 1: Build release installers
1. From project root run:
2. npm run release:windows
3. npm run release:mac
4. npm run release:linux

### Step 2: Open GitHub release screen
1. Go to your repository on GitHub.
2. Click Releases on right side.
3. Click Draft a new release.
4. Choose or create tag, example: v0.1.0.
5. Enter title, example: STU-LS v0.1.0.
6. Add notes (what changed).

### Step 3: Upload release assets
1. Upload installer files.
2. Upload updater artifacts generated by Tauri (including latest.json and signature files).
3. Click Publish release.

### Step 4: Verify update metadata URL
1. Open in browser:
2. https://github.com/OWNER/REPO/releases/latest/download/latest.json
3. Confirm it is reachable.

Why:
1. If this URL fails, in-app updater cannot find updates.

## 3. Every Future Update Cycle

### Step 1: Make code changes and push
1. Commit and push your changes.

Important:
1. Push alone does not update user apps.

### Step 2: Increase version
1. Update version in [apps/desktop/package.json](apps/desktop/package.json).
2. Update same version in [apps/desktop/src-tauri/tauri.conf.json](apps/desktop/src-tauri/tauri.conf.json).

Example:
1. Old: 0.1.0
2. New: 0.1.1

### Step 3: Build and publish new GitHub Release
1. Run release build commands.
2. Create new release tag, example v0.1.1.
3. Upload all new release assets.
4. Publish release.

### Step 4: User update experience on laptops
1. User opens app.
2. User clicks Check for Updates.
3. If update exists, app shows Download and Install.
4. User clicks Restart to Apply Update.

## 4. How to Use Your Own Domain for Installer Link

You asked for links like:
1. https://mydomain.com/desktop_app.exe

This is possible in two secure ways.

### Option A: Host installer directly on your server/CDN (best for custom branding)
1. Upload installer file to your server storage.
2. Serve it at stable URL:
3. https://mydomain.com/downloads/desktop_app.exe
4. Use this URL on your website button.

### Option B: Keep GitHub as storage but hide link behind your domain
1. Create your own route, example:
2. https://mydomain.com/download
3. Your backend redirects to latest installer asset URL.
4. Users only see your domain link.

Note:
1. Website download link and in-app update channel can be different.
2. Common setup:
3. Website first install via your domain URL.
4. In-app updates via GitHub latest.json.

## 5. How It Connects to User Laptops
1. App already has updater endpoint in config.
2. On Check for Updates, app calls that endpoint.
3. Endpoint returns latest version metadata.
4. App compares current app version vs latest version.
5. If newer, app downloads signed update artifact for OS/arch and installs.

## 6. Common Mistakes (Avoid)
1. Forgot to bump version number -> app says no updates.
2. Uploaded installer but not updater metadata/signatures -> check/install fails.
3. Wrong public key in config -> signature validation fails.
4. Changed private key after releasing app -> old installed clients cannot trust new updates.

## 7. Quick Checklist Before Every Release
1. Version bumped in both files.
2. Signing key env vars set.
3. Installers and updater assets generated.
4. New GitHub release drafted and assets uploaded.
5. latest.json URL reachable.
6. Test update from one old installed app.
